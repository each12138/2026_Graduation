# 复杂指令支持完善建议

## 1. 当前实现的能力边界

当前程序更接近一个“单轮决策、单技能执行”的控制器，而不是一个完整的任务规划与执行系统。

现有流程可以概括为：

1. 用户输入一句指令。
2. 程序读取当前机器人状态。
3. 将当前状态、用户指令、语义地图和历史记录拼接成 Prompt。
4. 让 LLM 只选择一个技能。
5. 调用对应技能执行一次。
6. 返回结果并等待下一条输入。

因此，当前系统适合处理这类简单命令：

- 站起来
- 坐下
- 打招呼
- 前进
- 记录当前位置
- 去某个已保存地点

但它并不具备稳定处理复杂复合任务的能力。

## 2. 当前实现存在的核心限制

### 2.1 LLM 只能选择一个动作

当前 Prompt 明确要求模型：

- 只能从技能列表中选择一个动作
- 输出单个 JSON 对象

这意味着像“先去地点1，再打招呼，然后回来”这样的语句，即使语义上包含多个步骤，模型最终也只能返回一个动作，例如：

- 只返回 `navigate(location=地点1)`
- 或只返回 `hello`

系统本身没有“动作序列”这个概念。

### 2.2 没有任务规划能力

当前 `history` 只是供 LLM 参考的上下文，并不是“待执行计划”。

程序内部没有：

- 任务列表
- 子任务队列
- 当前执行到第几步
- 剩余步骤
- 失败后的补救路径

所以即使 LLM 知道用户想做多步任务，系统也没有地方保存和推进这些步骤。

### 2.3 导航只发目标，不等待完成

当前 `navigate` 的实现只是：

1. 从语义地图中读取目标点位姿。
2. 通过 TCP 把目标点发送给导航端。
3. 立即返回“目标已发送”。

它不会：

- 等待机器人真正到达
- 判断导航是否成功
- 检查是否超时
- 获取失败原因
- 在到达后自动执行下一步

因此所有“先导航，再执行动作”的复合命令都无法可靠串起来。

### 2.4 没有“返回此处”所需的临时位置记忆

当前系统只有显式的 `memory_position(name, pose)`。

这意味着：

- 用户如果说“记住这里”，可以保存当前位置
- 用户如果直接说“去地点1然后返回此处”，系统并不会自动把“此处”记录为一个临时锚点

所以“返回此处”这种表达在当前架构下通常无法闭环完成。

### 2.5 没有持续任务机制

像“在地点1、地点2、地点3之间持续巡航”这种任务，本质上不是一次性动作，而是一个持续运行的行为。

当前程序没有：

- 后台执行循环
- 运行中任务状态
- 当前巡航目标
- 循环策略
- 停止/取消命令

主程序执行完一次动作后就会重新阻塞在 `input()`，等待用户下一条输入，因此无法自然支持持续巡航。

### 2.6 缺少动作完成条件与失败处理

当前系统没有统一定义每个动作的：

- 前置条件
- 完成条件
- 超时条件
- 失败条件
- 重试策略

例如：

- `hello` 何时算结束
- `navigate` 何时算到达
- 导航失败后是重试、跳过还是终止

这些都没有结构化表达。

## 3. 复杂指令下可能出现的问题

### 3.1 示例一：移动到地点1并打招呼然后返回到此处

这类指令的理想执行链路应为：

1. 保存当前所在位置为临时原点。
2. 导航到地点1。
3. 等待确认到达地点1。
4. 执行打招呼动作。
5. 导航回原点。
6. 等待确认返回完成。

但在当前实现下，常见问题包括：

- LLM 只能选一个动作，无法直接输出完整计划。
- 系统不会自动保存“此处”为临时位置。
- 即使第一步选中了 `navigate`，程序也只会发送导航目标，不会等待到达。
- 到达后不会自动执行 `hello`。
- 更不会自动继续执行返回动作。

最终结果通常会退化为：

- 只去地点1
- 只打招呼
- 或者因地点不存在而直接报错

### 3.2 示例二：在地点1、地点2、地点3之间进行持续巡航

理想链路应为：

1. 解析出巡航地点列表。
2. 建立循环任务。
3. 导航到地点1并等待到达。
4. 再去地点2并等待到达。
5. 再去地点3并等待到达。
6. 按策略继续循环，直到收到停止命令。

但当前实现中会遇到的问题包括：

- 系统无法把一句话拆成地点列表和循环任务。
- 没有任务队列保存后续导航目标。
- 没有导航完成反馈，因此不知道何时切换到下一个点。
- 没有后台状态机支持“持续执行直到取消”。
- 没有“停止巡航”这类运行时控制接口。

所以它最多只能：

- 选择其中一个地点执行一次导航
- 或因为指令过于复杂导致模型输出不稳定

## 4. 建议的完善方向

### 4.1 从“单动作决策”升级为“任务规划 + 执行器”

建议不要再让 LLM 直接只输出一个动作，而是让它输出一个结构化计划。

例如：

```json
{
  "task_type": "sequence",
  "steps": [
    {"action": "remember_current_position", "name": "__origin__"},
    {"action": "navigate", "location": "地点1"},
    {"action": "hello"},
    {"action": "navigate", "location": "__origin__"}
  ]
}
```

这样 LLM 的职责变成“拆解任务”，而不是“立即决定唯一动作”。

### 4.2 增加任务队列和状态机

系统内部应维护结构化任务状态，例如：

- 任务 ID
- 任务类型
- 步骤列表
- 当前步骤索引
- 每步状态：`pending / running / succeeded / failed`
- 整体任务状态：`idle / running / paused / completed / failed / cancelled`

这样在接到复杂命令后，系统就可以自主推进，而不是每执行一步就回到输入阻塞状态。

### 4.3 为导航增加完成反馈

这是复杂任务支持中最关键的一项。

导航模块应补充以下能力：

- 发送目标后等待导航结果
- 判断是否到达目标点
- 设置到达距离阈值
- 设置导航超时
- 返回成功、失败、取消等状态
- 失败时给出原因

只有导航真正具备“开始 - 运行中 - 到达/失败”的完整生命周期，系统才能稳定支持“先去 A，再做 B”。

### 4.4 自动支持“返回此处”这类临时引用

建议引入临时位置记忆机制。

当用户的指令中出现：

- 返回此处
- 回到这里
- 回原位

系统应在任务开始前自动记录当前位置，例如保存为：

- `__origin__`
- `__start_point__`

之后在计划中直接引用该临时点位，而不要求用户先手动执行一次“记住这里”。

### 4.5 支持复合语义解析

复杂指令一般包含多种语义关系，建议让上层解析模块支持以下类型：

- 顺序关系：先 A 再 B 然后 C
- 循环关系：在 A、B、C 之间循环
- 条件关系：到达后执行某动作
- 持续关系：持续执行直到停止
- 指代关系：这里、此处、刚才的位置、上一个地点

这一步可以由 LLM 负责自然语言理解，但输出必须是结构化结果，而不是自由文本。

### 4.6 区分一次性任务和持续性任务

建议将任务分成两大类：

- 一次性任务
- 持续性任务

一次性任务示例：

- 去地点1打招呼后回来
- 去地点2坐下

持续性任务示例：

- 在地点1、地点2、地点3之间持续巡航
- 在大厅持续巡视直到停止

持续性任务需要额外支持：

- 后台循环执行
- 当前循环轮次
- 停止/取消/暂停/继续

### 4.7 为每个技能补齐前置条件和后置条件

建议给技能增加统一元信息，例如：

- 前置条件
- 参数校验
- 完成条件
- 超时条件
- 失败处理方式

例如：

- `navigate(location)` 的完成条件是“到达目标点阈值范围内”
- `hello()` 的完成条件是“动作执行结束”
- `sit_down()` 的前置条件可能是“机器人当前处于可安全坐下状态”

这样执行器才能更可靠地串联技能。

### 4.8 将历史记录升级为任务记忆

当前 `history` 更像对话上下文，不适合作为复杂任务的执行依据。

建议新增独立的任务上下文存储，用于记录：

- 当前任务目标
- 当前步骤
- 已完成步骤
- 临时变量
- 运行中状态
- 失败原因

LLM 只负责：

- 初次规划
- 失败后的重规划
- 不确定场景下的解释

而不是负责记住所有执行细节。

## 5. 推荐的最小可行改造顺序

如果希望以较小代价逐步升级，建议按以下顺序推进：

1. 让 LLM 输出“步骤列表”，替代单个动作。
2. 增加本地任务队列，支持顺序执行多个步骤。
3. 给 `navigate` 增加等待到达和结果反馈。
4. 增加临时原点保存，支持“返回此处”。
5. 增加持续任务执行循环和停止接口。
6. 再进一步补充失败重试、任务恢复和重规划能力。

## 6. 总结

当前代码已经能够完成“自然语言到单个机器人技能”的映射，适合作为简单口令控制原型。

但如果要支持如下复杂命令：

- 移动到地点1并打招呼然后返回到此处
- 在地点1、地点2、地点3之间进行持续巡航

则必须把系统从“单步动作触发器”升级为“任务规划与状态驱动执行系统”。

核心改造重点可以概括为四点：

- 让 LLM 输出计划，而不是单个动作
- 在本地维护任务队列和运行状态
- 为导航提供完成反馈
- 支持临时记忆点与持续任务控制

只有这样，机器人才能稳定处理多步骤、可循环、可中断的真实复杂指令。
